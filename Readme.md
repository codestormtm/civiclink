# CivicLink

CivicLink is a multi-role civic complaint management platform for Sri Lanka. It lets citizens report local issues, helps department administrators triage and assign work, gives field workers a task portal, and exposes public transparency data for resolved and in-progress civic services.

The project is organized as one Node.js API plus three separate React portals:

- `web-public`: citizen portal for registration, login, manual complaint submission, AI-guided complaint intake, complaint tracking, settings, and the public transparency dashboard.
- `web-admin`: system admin and department admin portal for departments, issue types, complaint queues, worker management, reports, password-reset requests, and system monitoring.
- `web-worker`: worker portal for assigned tasks, status updates, evidence uploads, language preferences, and live task notifications.
- `backend`: Express API, PostgreSQL access, MinIO attachment storage, Socket.IO realtime events, role-based authorization, Firebase session exchange, and AI-guided intake.
- `mobile/citizen` and `mobile/worker`: Android-first Flutter WebView shells for LAN testing the citizen and worker portals with native Firebase Cloud Messaging registration.

## Project Purpose

CivicLink is intended to reduce friction between citizens and local government teams by:

- Capturing structured civic complaints with location data and attachments.
- Routing complaints to the correct department and issue type.
- Supporting role-specific workflows for citizens, system admins, department admins, and workers.
- Tracking complaint status from submission through assignment, work, resolution, rejection, and closure.
- Providing public transparency views for department performance and complaint activity.
- Supporting multilingual citizen and worker experiences.
- Offering AI-assisted complaint intake through Groq when configured.

## Main Features

- Citizen account registration and login.
- Optional Firebase citizen login/session exchange.
- Citizen preferred-language setup and settings.
- Manual complaint submission with department, complaint type, description, address, GPS/map location, and attachment support.
- AI-guided complaint reporting in English, Sinhala, and Tamil through Groq.
- Complaint tracking for citizens.
- Public dashboard with statistics, recent resolved complaints, department summaries, map points, comparisons, and public attachment viewing.
- System admin management for departments, complaint types, department admins, password-reset requests, and monitoring.
- Department admin complaint queue, worker assignment, reassignment, priority/SLA handling, wrong-department rejection, workload reporting, and performance reporting.
- Worker dashboard, task detail view, progress updates, resolution flow, field notes, evidence attachments, browser notifications, and live task updates.
- Socket.IO events for new issues, assignments, and status updates.
- MinIO-backed upload storage.
- PostgreSQL schema and seed scripts.
- Ubuntu deployment helper that installs dependencies, provisions infrastructure, builds frontends, and writes Nginx configuration.

## Roles

- `SYSTEM_ADMIN`: manages departments, complaint types, department admins, password reset requests, and system monitoring.
- `DEPT_ADMIN`: manages one department's complaint queue, workers, assignments, priority, SLA/reporting screens, and wrong-department rejections.
- `WORKER`: works assigned complaints, changes task status, adds notes, and uploads evidence.
- `CITIZEN`: submits complaints, uses AI-guided reporting, uploads attachments, and tracks complaint progress.

## Local Development Setup

Install dependencies for each workspace:

```bash
npm install
npm install --prefix backend
npm install --prefix web-public
npm install --prefix web-admin
npm install --prefix web-worker
```

Start infrastructure:

```bash
docker compose -f docker/docker-compose.yml up -d postgres redis minio
```

Docker Compose will use the defaults from `docker/docker-compose.yml`. You can also create `docker/.env` manually and pass it with `--env-file docker/.env`:

```env
SERVER_HOST=localhost
NODE_ENV=development
MONITORING_ENABLED=true
DB_USER=civic
DB_PASSWORD=civic123
DB_NAME=civiclink
DB_PORT=5433
JWT_SECRET=replace-with-a-long-random-secret
JWT_EXPIRES_IN=1d

PUBLIC_PORT=5173
ADMIN_PORT=5174
WORKER_PORT=5175
MINIO_PORT=9000
MINIO_CONSOLE_PORT=9001
MINIO_ACCESS_KEY=admin
MINIO_SECRET_KEY=password123

```




## Ubuntu Deployment

The production-style installer is `scripts/install-ubuntu-server.sh`. It:

- Installs required Ubuntu packages.
- Installs Node.js 22 when needed.
- Installs Docker Engine and the Docker Compose plugin when needed.
- Enables Docker and Nginx.
- Writes `backend/.env`, `docker/.env`, and frontend production env files.
- Starts PostgreSQL, Redis, and MinIO.
- Applies database schema and migrations.
- Builds and starts the backend container.
- Creates the initial system admin if needed.
- Installs and builds all three frontend apps.
- Writes an Nginx config for the public, admin, and worker portals.
- Opens UFW ports when UFW is active.

Prepare deployment values:

```bash
cp scripts/ubuntu-install.env.example scripts/ubuntu-install.env
nano scripts/ubuntu-install.env
```

Run the installer from the repository root:

```bash
sudo bash scripts/install-ubuntu-server.sh
```

Optional custom config path:

```bash
sudo CONFIG_FILE=/path/to/civiclink.env bash scripts/install-ubuntu-server.sh
```

After a successful install, the script prints the portal URLs, backend health URL, MinIO URLs, database details, and initial system admin credentials.

## Nginx Routing Model

The Ubuntu installer creates three Nginx server blocks:

- Public/citizen portal on `PUBLIC_PORT`, default `5173`.
- Admin portal on `ADMIN_PORT`, default `5174`.
- Worker portal on `WORKER_PORT`, default `5175`.

Each server block:

- Serves the matching Vite `dist/` directory.
- Proxies `/api/` to the backend.
- Proxies `/socket.io/` to the backend with WebSocket upgrade headers.
- Falls back to `index.html` for client-side routing.

## Important API Areas

- `/api/auth`: registration, login, worker login, current session, preferences, Firebase session exchange, and forgot-password flows.
- `/api/citizen-complaints`: citizen complaint creation, tracking, department/type lookup, and attachments.
- `/api/intake`: AI-guided complaint intake sessions and final submission.
- `/api/issues`: complaint list, map data, status changes, and priority changes.
- `/api/assignments`: department admin assignment actions.
- `/api/worker`: worker assignment list, detail, status updates, and evidence attachments.
- `/api/workers`: department worker management.
- `/api/departments`: department and issue type management.
- `/api/dept-admin`: department summary, queue, workload, and reports.
- `/api/system-admin`: department admin creation, password reset requests, monitoring summary/incidents/logs/activity/alerts.
- `/api/public`: public statistics, resolved complaints, department summaries, map points, comparisons, and public attachment viewing.
- `/api/health/app`: backend health.
- `/api/health/db`: database health.
- `/api/health/storage`: MinIO health.

## Realtime Events

Socket.IO authenticates with the same JWT used by the API. The backend emits events such as:

- `new_issue`
- `task_assigned`
- `status_updated`

The admin and worker portals connect to Socket.IO to refresh dashboards and show task/update notifications.

## Monitoring

When `MONITORING_ENABLED=true`, the backend starts a monitoring loop after startup. Monitoring configuration includes:

- Admin portal URL
- Worker portal URL
- Citizen portal URL
- Transparency portal URL
- Backend API health URL

System admins can access monitoring summaries, incidents, logs, department activity, worker activity, and alerts through `/api/system-admin/monitoring/*` routes.

## AI Intake

AI-guided reporting is optional. Set `GROQ_API_KEY` in `backend/.env` to enable it.

The current AI service uses:

- Provider: Groq
- Model: `llama-3.3-70b-versatile`
- Languages: English, Tamil, Sinhala

If `GROQ_API_KEY` is missing, AI intake endpoints return a service-unavailable response while the rest of the platform can still run.

## Firebase Auth

Firebase is optional and is used for citizen Firebase session exchange.

Backend options:

- `FIREBASE_SERVICE_ACCOUNT_JSON`
- Or `FIREBASE_PROJECT_ID`, `FIREBASE_CLIENT_EMAIL`, and `FIREBASE_PRIVATE_KEY`

Frontend public portal options:

- `VITE_FIREBASE_API_KEY`
- `VITE_FIREBASE_AUTH_DOMAIN`
- `VITE_FIREBASE_PROJECT_ID`
- `VITE_FIREBASE_STORAGE_BUCKET`
- `VITE_FIREBASE_MESSAGING_SENDER_ID`
- `VITE_FIREBASE_APP_ID`

If Firebase web values are missing, standard local citizen login can still be used, but Firebase login will not be available.

## Known Notes

- There is no automated test suite currently defined in the repository.
- The backend hard-fails on missing required environment variables.
- `backend/sql/001_schema.sql` and later SQL files should be applied together for a complete current database.
- The SLA migration is implemented as `backend/scripts/addSlaFields.js`.
- Redis is included in Docker Compose but is not currently required by the application code.
- Frontend apps are separate Vite projects, not a single monorepo workspace package.
- Root helper scripts are mainly Windows-focused; use per-project commands on Linux/macOS.
